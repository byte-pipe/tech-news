### zortaxcommentedJan 14, 2026•edited

The blade graphics library is a mess and causes several issues for both Zed users as well as other 3rd party apps using GPUI. This PR removes blade and implements the linux platform usingwgpuwhich is the de-facto standard in the rust UI and graphics ecosystem. This will not justfixissues that Zed users have today, but also profit from wgpu improvements in the futures, from other projects contributing (such as the bevy game engine, Iced, or pretty much every other relevant project).

This will close several related issues on the zed repo as well. Seehttps://github.com/zed-industries/zed/issues?q=frozen%20nvidia%20linux(probably not all of them, have only tested the freeze on nvidia and Smithay-based wayland compositors).

Some related issues:#44814#40481niri-wm/niri#2335zortax/zlaunch#15

Would appreciate feedback if this is something the zed maintainers would be interested in.

### zortaxcommentedFeb 5, 2026•edited

it works worse for vram and perfomance

have you read the thread above? This statement is simply not true when comparing to blade. The implementation already has a lower VRAM and memory footprint than the blade implementation had, and that is before some performance improvements addressing the comments above. Performance will not change with this PR significantly either, the wgpu implementation does essentially the same thing as the blade implementation did. Feel free to run benchmarks on this...

slow compilation and twice rust-analyzer ram usage at dev time

Have you tried compiling the zed repo? Wgpu build times are not the issue or bottleneck here. On my system the difference was barely noticable, and since we have incremental builds for dependencies most of it will be one-time only anyway.

sometimes it has long-time problems without fixes

Sure, wgpu has issues too. The difference is that it is actively maintained and a core dependency of many big projects. There is generally a big interest in getting significant things fixed. Blade on the other hand has open PRs (not even just issues) with small fixes that have a big impact (breaking issues) for downstream apps that haven't been merged in half a year. I don't want to attack the author, I know maintaining OSS is a lot of work and usually doesn't pay the bills, but zed and gpui are projects people rely on, so I don't think this is acceptable for a core dependency.

You personally like wgpu, i personally dislike it especially after development and usage of apps based on it

I could also turn that argument around, but more importantly I think community support and stability are plainly more important that slight differences in API ergonomics.

looks like something that can be made selectable by cfg flags based on checking this pr

Yes of course it could, but that has certain implications that should be considered. If you add it as a new backend, it will also have to be maintained, so more work for further improvements. This is already a bit of an issue with GPUI given the different renderer implementation. I respect the decision of the zed maintainers to support those implementation, but personally I think we should thrive to consolidate the rendering infrastructure, not further fragment it. There are exciting things for the future that could be enabled (or significantly simplified for end-users) by consolidation, like APIs for custom shaders, direct buffer access for custom elements, embedding into other wgpu apps (or being embeddable by them), like bevy.
