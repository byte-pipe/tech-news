# Why I Write Games in C (yes, C)

This article explains the author's preference for using 'vanilla' C for solo game development, a relatively uncommon practice. The author outlines the key requirements and desires they have from a programming language and then discusses alternative languages, ultimately concluding that C remains the best fit for their needs.

## What I need from a language

The author prioritizes reliability, wanting to avoid bugs they didn't create and ensuring the longevity of their projects by choosing a portable language suitable for various platforms, including consoles. Portability and good library support are crucial.

## What I want from a language

Simplicity is a strong desired feature, with the author preferring a language that is easy to learn and remember. They also value features that facilitate bug prevention, such as strict typing, strong warnings, and static code analysis, alongside good debugging tools. Performance is important for expanding creative possibilities, and fast compilation speeds are essential for maintaining workflow. The author is not an OOP convert and prefers to handle data and code independently.

## The Alternatives

The author discusses several alternatives to C, including C++, C#, and Java, finding them too complex, verbose, or overly prescriptive in their OOP approach. Go is appealing but faces drawbacks like garbage collection and limited game library support. The author is wary of rapidly evolving web technologies like JavaScript and finds Haxe more promising but questions its long-term viability. The idea of creating a custom language is considered but deemed too burdensome.

## Why C is still my best fit

Despite being potentially dangerous, C is praised for its reliability and simplicity. It offers exceptional speed and compilation times and boasts strong, ongoing library and tooling support, making it highly portable across different platforms. The author emphasizes that this is a personal preference and not a recommendation for others to use C.
