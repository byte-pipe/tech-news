# 5 Myths About Legacy Code You Should Stop Believing - DEV Community

Here's a concise and informative summary of the article in Markdown format:

## Debunking Common Myths About Legacy Code

* **Myth 1:** "Legacy code can only be ancient or exotic." Reality: it can be slightly outdated, like an Angular version that still works.
* **Myth 2:** "Legacy is the previous team's fault." Myth-shattering fact: often, nobody knows the full story behind inherited code, and blaming the past owner is not as productive as understanding context.
* **Myth 3:** "Migrating legacy code is a developer's obsession." Reality: migrating should be seen as risk management to address security issues and performance problems, not a fun project with no constraints.
* **Myth 4:** "Big-bang rewrite is always an antipattern." Myth-rebuking fact: sometimes a clean rewrite is the fastest and most efficient solution for small, isolated apps that are not mission-critical.
* **Myth 5:** "The strangler pattern is the only correct approach to migrating legacy code." Reality: this migrates large, complex applications slowly and risks involving two technologies at once. Sometimes you need a hybrid migration strategy.

## Avoiding Common Pitfalls in Legacy Code Migration

To avoid common pitfalls like age being the only factor that determines legacy status and overanalyzing the past owners:
- Take an active interest in understanding the context behind inherited code.
- Don't jump on migrating, as it may not always be necessary or even suitable for your project.
- Assess migration risks carefully to make informed decisions.
- Be flexible and willing to iterate with new technologies during the migration process.
