# 5 Myths About Legacy Code You Should Stop Believing - DEV Community

## Debunking Legacy Code Myths for Devs
=====================================================

As developers, it's easy to get caught up in myths about legacy code. However, these urban legends can hinder our ability to make informed decisions when migrating old frontends into modern tools.

### Myth 1: "Legacy means ancient or exotic."

Not necessarily. Sometimes it's just a slightly outdated technology or software that still works for now but will need significant changes or removals in the future.

* Examples might include:
	+ Using an Angular version from 2015
	+ Deploying a React-based application with class components
	+ Sticking with Webpack 3 due to community support

Legacy code can be old news; it's not automatically exotic.

### Myth 2: "Legacy is the previous team's fault."

Blaming people can lead to delays and frustration. It's essential to understand the context behind the code you're inheriting.

* What might have happened in the past:
	+ Projects were frozen for an extended period
	+ The original team may not have the necessary experience or time to migrate

Understanding context is key to resolving these myths.

### Myth 3: "Migrating legacy is just a developer's obsession."

Legacy code can silently harm business operations, security, performance, and more. A migration should be informed risk management rather than simply a project to impress.

* Potential issues:
	+ Security threats
	+ Performance challenges
	+ High maintenance costs

Business and team welfare come first; migrating legacy shouldn't be an exercise in vanity.

### Myth 4: "Big-bang rewrites are always the best approach."

This is often not true, especially for small applications or those without critical mission-critical requirements. Big-bang migrations can be overrated and cause more harm than good.

* Pros:
	+ Simpler
	+ Safer
	+ Quicker solutions

Consider when migrating legacy code; a well-considered approach may yield unexpected benefits.

### Myth 5: "Strangler migration is always the correct approach."

Strangler migrations can be problematic, especially for large applications with complex requirements. There's no one-size-fits-all solution; choose the hybrid that works best.

* Considerations:
	+ Time
	+ Team knowledge
	+ Boundary management

The goal is to strike a balance between legacy code preservation and modern adaptation.

By recognizing these myths, developers can take steps towards more accurate migration strategies.
