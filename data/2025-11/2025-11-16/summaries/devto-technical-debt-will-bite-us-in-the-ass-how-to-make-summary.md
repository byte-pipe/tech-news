# "Technical Debt Will Bite Us in the Ass": How to Make Non-Technical Stakeholders Actually Care - DEV Community

**Summary**

The article "Technical Debt Will Bite Us in the Ass: How to Make Non-Technical Stakeholders Actually Care" explores the challenges of communicating technical debt to non-technical stakeholders. The author shares a personal story of how they struggled to explain code quality issues to product managers, who were focused on delivering features rather than building high-quality architecture.

The key takeaway is that technical jargon and terminology can create communication barriers, leading stakeholders to dismiss critical issues. The article argues that using language such as "refactoring," "technical debt," and "architecture" can be misleading for non-technical stakeholders, who focus on short-term goals like features shipped and revenue generated.

**Main Points**

* Technical debt is not an excuse for poor code quality
* Non-technical stakeholders often prioritize features over architecture
* Using technical jargon or terminology can create communication barriers
* Refactoring refers to refactoring the existing architecture or code, which is irrelevant to non-technical stakeholders
* The goal should be to explain how technical issues affect overall system health and efficiency

**Key Messages**

* Use language that resonates with stakeholders from different departments (e.g., features, revenue, customer satisfaction)
* Be precise in explaining technical concepts, avoiding abstract terms like "architecture"
* Focus on the impact of technical debt on the entire system, rather than just focusing on individual code changes
* Consider how to translate technical jargon into language that non-technical stakeholders can understand
