# Claude Code Is Not Making Your Product Better

## K‑shaped productivity curve
- Data from labor economists shows AI coding agents boost output for senior engineers but not for junior engineers.  
- Senior output has risen since the 2023 LLM inflection; junior output is flat or declining.  
- The “K‑shape” means productivity gains are unevenly distributed across seniority.

## What top engineers are observing
- Recent posts from Dax (buildingopencode.ai), Karri Saarinen (Linear), and David Cramer (Sentry) reveal a common pattern despite their support for AI.  
- Dax admits his team is still searching for effective ways to use coding agents.  
- Karri’s “drunken whiteboard” reply highlights uncertainty about AI‑driven speed gains.  
- Cramer argues that LLMs remove the barrier to start but introduce complexity and bloat, slowing long‑term velocity.  
- All three note difficulty in accelerating product‑improvement velocity with current coding agents.

## Why Claude Code isn’t a game‑changer
- Claude Code is “Claude‑coded,” so if it truly accelerated product improvement, the advantage should compound exponentially (e = e^(c·x)).  
- In practice, competitors like Codex, Cursor, Cognition, and Factory remain viable, and the gap has not widened.  
- Possible explanations: gains are eaten by complexity debt, or Anthropic’s large engineering team dilutes marginal per‑engineer benefit.  
- The core bottleneck is not code production speed but something harder to quantify.

## Lines of code as a cost, not a product
- Mature engineering cultures treat lines of code as a liability to be spent wisely, not a metric of productivity.  
- Each line adds surface area for bugs, dependencies, and future work; complexity compounds non‑linearly.  
- Examples: TinyChat’s alarm for codebase size, Linear’s high quality with a small codebase versus Jira’s massive effort and lower quality.  
- Large organizations (e.g., Facebook) are limited not by UI code speed but by the need to compress functionality into minimal, reliable code that scales to billions.  
- The true reward function for engineering is compression and maintainability, not raw lines of code produced.
