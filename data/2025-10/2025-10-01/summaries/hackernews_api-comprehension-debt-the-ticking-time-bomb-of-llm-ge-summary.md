# Comprehension Debt: The Ticking Time Bomb of LLM-Generated Code – Codemanship's Blog

## The Comprehension Debt Problem with Large Language Models

The use of Large Language Models (LLMs) has led to an increase in time it takes developers to modify or fix code generated by these models. This phenomenon is not new, as developers have been experiencing this issue for decades when working on legacy systems.

### Understanding the Issue

Before making changes to code, there is typically a need to understand what the code does and why it was created in the first place. However, when dealing with rapidly generated code using LLMs, this process can be time-consuming due to:

*   The sheer amount of unseen code being added
*   The difficulty in comprehending complex algorithms or data structures

### Solved Approach and Alternatives

Teams that are particularly quick to adapt have started focusing on a different approach - reviewing and understanding the generated LLM-generated code before deploying it. This method has gained attention, but there is also anecdotal evidence of teams opting for other strategies:

*   **Code Review**: Teams that care about quality often spend more time reviewing and understanding original codebases before integrating new technology.

### The Comprehension Debt

When software use leads to the generation of unused or incomplete code, creating a problem known as "comprehension debt". This often leads to downstream effects where generated code requires modification, potentially slowing down development efforts over time.
